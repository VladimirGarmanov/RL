# Максимальная плавность и активный баланс

## Проблема: "нужно плавнее"

Движения бывают рывкообразные, особенно при смене команды.

## Главный принцип: плавность — это награда, а не оптимизатор

Ранняя версия пыталась получить плавность, задушив сам PPO:
`lr=5e-5, clip=0.1, target_kl=0.02, 180М шагов`. Это лечило симптом:
обучение шло неделю, эксплорация схлопывалась, а рывки всё равно
определялись тем, за что среда наказывает.

Правильное разделение труда:
- **Среда** (centipede_env.py) — говорит, ЧТО такое плавно: штрафы на
  рывки действий, размах суставов, моменты + НЧ-фильтр действий.
- **Оптимизатор** (train.py) — учится быстро со стандартными
  гиперпараметрами и затухающим learning rate.

## Оптимизатор: стандартные значения + затухание

```python
learning_rate = linear_schedule(3e-4, 3e-5)  # быстрый старт, тонкая полировка в конце
clip_range = 0.2         # стандарт PPO
target_kl = 0.05         # мягкий предохранитель, не ручной тормоз
ent_coef = 0.001         # std не схлопывается раньше времени
n_steps = 2048           # 8 сред × 2048 = 16384 примеров на обновление
batch_size = 1024
net_arch = [512, 256]    # ~196k весов — влезает в Teensy/ESP32
COMMAND_TIMESTEPS = 40_000_000  # ~18–24 часа на 8-ядерном CPU
```

Затухание lr даёт тот же эффект «микроскопических шагов в конце», что и
lr=5e-5, но не заставляет платить за него всю дистанцию обучения.

## Плавность в centipede_env.py: усиленные штрафы

### 1. Штраф на рывки действий
```python
action_rate_weight = 0.20  # было 0.05 (в 4 раза больше!)
```

**В reward функции:**
```
reward -= 0.20 * sum((action_t - action_{t-1})²)
```

Каждый шаг при смене действия на >0.1 коэффициентов = минус 0.05 к награде.
Политика учится менять действия как можно плавнее, постепенно.

### 2. Штраф на размахивание суставами
```python
dof_vel_cost_weight = 5e-4  # было 1e-4 (в 5 раз больше)
```

**В reward функции:**
```
reward -= 5e-4 * sum(joint_velocity²)  # все 23 степени свободы
```

Суставы двигаются быстро = минус. Медленные, плавные движения = плюс.

Это заставляет политику двигаться, как если бы на каждый сустав была пружинка.

### 3. Усиленный штраф на момент (крутящий момент)
```python
torque_cost_weight = 0.05  # было 0.02
```

**Суть:** резкие рывки требуют больших моментов на валах.
Политика учится крутить сервоприводы плавно, не со взрывной силой.

## Активный баланс: даже на месте работает

Когда робот стоит (`v_x=0, w_z=0`), обычно просто "окаменевает" — замирает.
С текущими штрафами на размахивание суставов он будет:

- Микро-регулировать позицию центра масс
- Слегка раскачиваться взад-вперёд для баланса
- Напрягать мышцы (в смысле моторчиков) чтобы не упасть от толчков

Это активный баланс — робот "дышит" ногами, слегка крутя суставами, чтобы остаться на ногах.

## Чего ожидать на разных этапах обучения

| Шаги | Поведение |
|------|-----------|
| 0–3М | Хаотичные движения, часто падает |
| 3–10М | Ходит, но рывкообразно, падает редко |
| 10–25М | Ходит плавнее, баланс улучшается, проявляется волна ног |
| 25–40М | Плавная метахрональная походка, lr затух — полировка |

## Когда остановиться?

Смотрите tensorboard/логи: если `ep_len_mean` вышел на плато у максимума и
разбивка награды (`reward_gait_coordination`, `cost_action_rate` в info)
стабилизировалась — можно останавливать раньше:
```bash
# просто Ctrl+C, если удовлетворены результатом
# (сохранится в checkpoints/, потом load и play)
```

Если после 40М видно, что кривые ещё растут:
```bash
python train.py --reward-mode command --steps 60000000 --resume
```

## На реальный робот

Когда перенесёте на Teensy / ESP32:
1. Нормализация наблюдений (`vecnormalize_final.pkl`)
2. НЧ-фильтр действий (`action_filter_alpha = 0.3`)
3. Очень важно: **инверс НЧ-фильтра** в контроллере (замедлить PWM-сигналы)
4. На борту уже БУДЕТ плавность встроена в политику — дополнительные фильтры могут помешать

Поэтому учимся с фильтром в симуляции, переносим фильтр на робот, готово.
