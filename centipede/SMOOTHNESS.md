# Максимальная плавность и активный баланс

## Проблема: "нужно плавнее"

Даже с 18М шагов движения бывают рывкообразные, особенно при смене команды.
Нужна очень осторожная, консервативная политика.

## Решение в train.py: 180М шагов + супер-консервативные гиперпараметры

### 1. Огромное количество шагов
```python
COMMAND_TIMESTEPS = 180_000_000  # было 12М, это в 15 раз больше
```

**Почему:** с каждым шагом сеть учится чуть-чуть. 180М шагов = очень долгое, плавное обучение.
Последние 100М шагов будут "полировкой" — огромное количество малых улучшений.

### 2. Больше параметров в нейросети
```python
net_arch = [768, 512]  # было [512, 256]
```

**Вес параметров:** 392K → 650K (в 1.6 раза больше)
**Почему:** больше ёмкость = может выучить более тонкие, гладкие движения.

### 3. Огромные батчи для гладких обновлений
```python
n_steps = 4096       # было 2048
batch_size = 2048    # было 1024
```

**Результат на каждое обновление:** 8 сред × 4096 шагов = 32768 примеров, батч 2048
→ 16 обновлений на батч вместо 8 → более стабильная оценка градиента

### 4. Очень низкий learning rate
```python
learning_rate = 5e-5  # было 1e-4 (в 2 раза ниже)
```

**Шаг обновления:** каждое обновление весов — микроскопический шаг.
За 180М шагов эти микроскопические шаги накапливаются в отличную политику.

### 5. Жёсткие ограничители на шаг
```python
clip_range = 0.1         # было 0.15
max_grad_norm = 0.3      # было 0.5
target_kl = 0.02         # было 0.03
```

**Суть:** политика не может прыгать далеко за один шаг.
- `clip_range=0.1` → действие не может измениться больше чем на 10% от старого
- `max_grad_norm=0.3` → градиент обрезается, огромные скачки невозможны
- `target_kl=0.02` → слишком резкие обновления сразу останавливаются

## Плавность в centipede_env.py: усиленные штрафы

### 1. Штраф на рывки действий
```python
action_rate_weight = 0.20  # было 0.05 (в 4 раза больше!)
```

**В reward функции:**
```
reward -= 0.20 * sum((action_t - action_{t-1})²)
```

Каждый шаг при смене действия на >0.1 коэффициентов = минус 0.05 к награде.
Политика учится менять действия как можно плавнее, постепенно.

### 2. Штраф на размахивание суставами
```python
dof_vel_cost_weight = 5e-4  # было 1e-4 (в 5 раз больше)
```

**В reward функции:**
```
reward -= 5e-4 * sum(joint_velocity²)  # все 23 степени свободы
```

Суставы двигаются быстро = минус. Медленные, плавные движения = плюс.

Это заставляет политику двигаться, как если бы на каждый сустав была пружинка.

### 3. Усиленный штраф на момент (крутящий момент)
```python
torque_cost_weight = 0.05  # было 0.02
```

**Суть:** резкие рывки требуют больших моментов на валах.
Политика учится крутить сервоприводы плавно, не со взрывной силой.

## Активный баланс: даже на месте работает

Когда робот стоит (`v_x=0, w_z=0`), обычно просто "окаменевает" — замирает.
С текущими штрафами на размахивание суставов он будет:

- Микро-регулировать позицию центра масс
- Слегка раскачиваться взад-вперёд для баланса
- Напрягать мышцы (в смысле моторчиков) чтобы не упасть от толчков

Это активный баланс — робот "дышит" ногами, слегка крутя суставами, чтобы остаться на ногах.

## Чего ожидать на разных этапах обучения

| Шаги | Поведение |
|------|-----------|
| 0–10М | Хаотичные движения, часто падает |
| 10–30М | Ходит, но рывкообразно, падает редко |
| 30–60М | Ходит плавнее, баланс улучшается |
| 60–120М | Очень плавно, редкие рывки, хороший баланс |
| 120–180М | Максимальная плавность, активный баланс, идеально |

## Когда остановиться?

Если после 80–100 часов обучения уже видно хорошую плавность:
```bash
python train.py --reward-mode command --steps 120000000 --resume
# обучить ещё до 120М вместо полных 180М
```

Или даже:
```bash
# просто Ctrl+C, если удовлетворены результатом
# (сохранится в checkpoints/, потом load и play)
```

## На реальный робот

Когда перенесёте на Teensy / ESP32:
1. Нормализация наблюдений (`vecnormalize_final.pkl`)
2. НЧ-фильтр действий (`action_filter_alpha = 0.3`)
3. Очень важно: **инверс НЧ-фильтра** в седдевикасе (замедлить PWM-сигналы)
4. На борту уже БУДЕТ плавность встроена в политику — дополнительные фильтры могут помешать

Поэтому учимся с фильтром в симуляции, переносим фильтр на робот, готово.
