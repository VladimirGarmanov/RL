# Робот-многоножка — управляемая ходьба в 3D (MuJoCo + PPO)

Симуляция робота-многоножки в реальных габаритах (сервоприводы MG996R,
шасси-пластины, алюминиевые ноги — см. список компонентов в
`centipede_model.py`). Робот учится **отрабатывать команды скорости**:
ехать вперёд/назад, поворачивать, останавливаться и стоять неподвижно.
После обучения им можно рулить стрелочками, как радиоуправляемой машинкой.

Цель проекта — обучить походку в симуляции так, чтобы потом перенести её
на настоящего робота (sim-to-real).

## Если вы не знакомы с RL — два абзаца теории

**Обучение с подкреплением (RL)**: агент (нейросеть-«политика») на каждом
шаге получает наблюдение (показания датчиков), выдаёт действие (целевые
углы сервоприводов) и получает награду — число «насколько хорошо идут
дела». Никто не показывает роботу, КАК ходить; он пробует случайные
движения, а алгоритм постепенно усиливает те, что приносят больше награды.
Всё искусство — в конструкции награды (см. формулу ниже): что поощряешь,
то и получишь, включая непредусмотренные лазейки.

**PPO (Proximal Policy Optimization)** — стандартный алгоритм для обучения
ходьбе: улучшает политику маленькими осторожными шагами (не даёт одному
неудачному обновлению разрушить уже выученную походку) и использует
вторую сеть-«критика», оценивающую перспективность состояний, чтобы шум
случайных проб меньше сбивал обучение. Мы берём готовую реализацию из
библиотеки Stable-Baselines3.

## Быстрый старт

### Без Docker (локально)

```bash
# 1. Посмотреть на робота (случайные команды серво, без обучения)
python play.py --random

# 2. Проверить клавиши и механику без PPO-модели
python play.py --manual

# 3. Обучить простую плавную ходьбу вперёд (3 млн шагов, ~2 часа)
python train.py

# Смотреть результат: только вперёд, стрелки поворота не работают
python play.py --forward

# 4. РЕКОМЕНДУЕМЫЙ ПУТЬ: одна универсальная сеть (максимум плавности)
# Одна команда — 180 млн шагов обучения, примерно 80-100 часов
# (запустите в screen/tmux, иначе терминал забьётся на 4+ дня)
python train_overnight.py

# Смотреть результат: полный контроль стрелочками, ОЧЕНЬ плавная ходьба
python play.py

# Или робот сам катается по программе: вперёд -> разворот -> дуга -> назад
python play.py --auto
```

### Docker (упакованное окружение, нет проблем с зависимостями)

```bash
# Обучить в контейнере (все зависимости уже внутри, моды сохраняются в ./models)
docker build -t centipede-train .
docker run -v $(pwd)/models:/app/centipede/models centipede-train train_overnight.py

# Или через docker-compose (удобнее, логи пишутся в файл)
docker-compose build
docker-compose up train

# После обучения просмотреть результат (локально)
python play.py --auto
```

**Важно:** модели, обученные до изменения среды (другая `reward_version` в
`meta.json`), несовместимы: они видели другую физику и наблюдения, поэтому в
новой среде спотыкаются и падают. `play.py` такие модели пропускает с
предупреждением — их нужно переобучить.

## Во время обучения — что смотреть в логах

```
| rollout/                |            |
|    ep_len_mean          | 800        | ← это главное! >500 шагов = робот не падает
|    ep_rew_mean          | 145        | ← награда растёт со временем
| train/                  |            |
|    approx_kl            | 0.025      | ← <0.05 = обновления плавные, не агрессивные
|    entropy_loss         | -50        | ← исследование (отрицательное = норма)
|    explained_variance   | 0.92       | ← >0.8 = критик хорошо оценивает состояния
```

- **ep_len_mean < 100** — робот падает на первых шагах, что-то не так с наградой/средой
- **ep_len_mean > 800** — робот стоит или ходит стабильно, обучение идёт правильно
- **approx_kl > 0.1** — обновления слишком резкие, алгоритм разносит, снизьте `learning_rate` или `clip_range`

Всё в **одном окне** (3D-вид + панель статуса): раньше 3D-окно MuJoCo и пульт
pygame конфликтовали за фокус клавиатуры на macOS, и стрелки терялись.

Робот: **стрелки** — ехать/поворачивать (плавный разгон ~0.4 с), **пробел** —
стоп, **R** — рестарт, **Esc** — выход.

Камера: **мышь с зажатой ЛКМ** — вращать, **колесо** — зум, **WASD** —
панорама по плоскости, **F** — следовать за роботом вкл/выкл, **Home** — сброс.

## Два способа обучения: forward-only vs командный универсал

### Forward-only (`python train.py`)
Робот учится только **плавно идти вперёд** (команда всегда "(v_x, 0)", поворот игнорируется).
- **Плюсы:** быстро (3 млн шагов, 2–3 часа), лёгкая задача для сети
- **Минусы:** стрелки поворота/назад не работают, нужны отдельные 4 сети для полного контроля

Реимение: идти вперёд, не быстрее цели

### Командный универсал (`python train.py --reward-mode command`)
Одна сеть видит **команду** `(v_x, w_z)` на каждом шаге и отрабатывает её.
- **Плюсы:** полный контроль стрелочками (вперёд/назад/влево/вправо/стоп), одна сеть вместо 4
- **Минусы:** сложнее (18 млн шагов, 15–20 часов), но стоит того

**Рекомендуемый подход:** `python train_overnight.py` запускает `train.py --reward-mode command --steps 18000000`

Если хотите быстро протестировать механику — сначала `python train.py` на 3 млн шагов,
потом `python train_overnight.py` на 18 млн от нуля (или дольше, но полнее).

## Как робот учится идти вперёд (режим forward-only)

По умолчанию `train.py` учит только один навык: плавно идти вперёд. В этом
режиме нет случайных команд поворота, нет стоп-навыка и нет накопления курса:
политика всё время видит команду "вперёд" и получает простой сигнал качества.

```
reward = 4.0 * clip(v_x / 0.25, -1, 1)       # идти вперёд, не быстрее цели
       + 1.0 * exp(-ошибка_скорости² / 0.20) # бонус за скорость около 0.25 м/с
       + 0.1 * жив
       - 1.0 * боковая_скорость²
       - 0.4 * поворот_на_месте²
       - 1.0 * верт_скорость²                 # не скакать вверх-вниз
       - 0.03 * (крен' ² + тангаж' ²)         # не раскачиваться
       - 1.5 * наклон_сегментов               # корпус параллелен полу
       - 400 * (высота - 0.095)²              # не проседать на брюхо
       - 50 * перекрёст_ног
       - 0.02 * сумма((момент/макс)²)
       - 0.05 * сумма((команда - пред)²)      # плавность серво
       - 1e-4 * сумма(скорости_суставов²)
       - 8.0 * падение
```

Для просмотра такой модели используйте `python play.py --forward`: он постоянно
подаёт ту же команду вперёд, на которой сеть обучалась.

## Командный режим

В наблюдения политики входит **команда** `(v_x, w_z)` — целевая скорость
вперёд/назад и поворота. Во время обучения команда случайно меняется каждые
~3.6 секунды; чистые развороты и дуги составляют больше половины команд.

Ключевая идея награды: команда — это **единая цель-«твист»**, как у машины
на круговой развязке. Обе компоненты нормируются на свои максимумы и
отслеживаются одним экспоненциальным ядром + одним шейпингом вдоль
командного направления. Поэтому на дуге нельзя «добрать» награду скоростью
вперёд, игнорируя поворот: максимум — только на правильной дуге радиуса
`v/w`, с закручиванием на неё и выходом по текущему курсу. Скорость и курс
меряются по среднему курсу всех сегментов (курс одной головы виляет вместе
с волной позвоночника и шумит).

```text
u = (команда_v / v_макс, команда_w / w_макс)        # цель в нормированном виде
a = (факт_v / v_макс,  факт_w / w_макс)             # факт там же

reward = 4.0 * exp(-(|u - a|² + дрейф²) / 0.10)     # единое ядро слежения
       + 2.0 * dot(u, a) / |u|                      # шейпинг вдоль команды
       # если команда стоп и нет ошибки курса: 0.6 * exp(-скорости² / 0.04)
       + 0.1 * жив
       - 3.0 * max(0, 0.35|u| - dot(u,a)/|u|)       # простой при команде ехать
       - 1.0 * верт_скорость²                        # не скакать вверх-вниз
       - 0.03 * (крен' ² + тангаж' ²)                # не раскачиваться
       - 1.5 * наклон_сегментов                      # корпус параллелен полу
       - 400 * (высота - 0.095)²                     # держать номинальную высоту
       - 50 * перекрёст_ног                          # соседние ступни ближе 6 см
       - 0.02 * сумма((момент/макс)²)                # экономим ток и нагрев серво
       - 0.10 * сумма((команда - пред)²)             # плавность политики
       - 1e-4 * сумма(скорости_суставов²)            # не размахивать ногами
       - 8.0 * падение                               # разовый терминальный штраф
```

Плюс к штрафам за резкость: целевые углы серво проходят через НЧ-фильтр
(EMA, `action_filter_alpha=0.3`) — резкие скачки политики физически не
доходят до приводов, как рампа PWM на реальном контроллере.

Перекрёст ног: для каждой пары соседних ступней одной стороны считается
зазор в горизонтальной плоскости; всё, что меньше `leg_clearance` (6 см),
штрафуется квадратично — ноги учатся шагать волной, не задевая друг друга.

Эпизод завершается досрочно при падении (высота головы вне 6.5-20 см) или если
ЛЮБОЙ сегмент завалился набок. Падение дополнительно получает разовый штраф:
без него политика могла на несколько десятков шагов просаживать корпус почти
до пола, собирать шейпинг скорости и только потом завершать эпизод.

## Шероховатая поверхность

Пол — не идеальная плоскость, а поле высот (heightfield) 10 x 10 м с пологими
случайными буграми до 8 мм (`terrain_roughness` в `centipede_env.py`).
Зона старта в центре плоская и плавно переходит в неровности. У каждого из
8 параллельных процессов обучения — свой случайный рельеф, поэтому политика
не может подстроиться под конкретные кочки и учится ходить по «земле вообще» —
это важный шаг к переносу на реального робота. `terrain_roughness=0` вернёт
идеальный пол, `terrain_seed=N` зафиксирует рельеф.

## Что за XML в assets/

MuJoCo принимает описание мира в формате **MJCF** — это XML-документ, где
дерево `<body>` задаёт скелет робота (вложенность = «кость крепится к кости»),
внутри каждого body: `<joint>` — ось вращения, `<geom>` — физическая форма
с массой и трением. Отдельно `<actuator>` перечисляет приводы (у нас
`<position>` — позиционные серво: команда = целевой угол), `<asset>` —
текстуры и поле высот, `<default>` — общие параметры классов деталей.
Файл `assets/centipede_6seg.xml` **не пишется руками** — его генерирует
`centipede_model.py` из констант с размерами; менять нужно константы,
XML пересоздастся при следующем запуске.

## Интерфейс среды (= интерфейс будущего робота)

- **Действия (29)**: нормализованные целевые углы серво в [-1, 1] —
  ровно то, что поедет в PWM на железе.
- **Наблюдения (110)**: углы и скорости суставов (энкодеры серво), высота и
  ориентация головы (IMU), предыдущая команда серво, текущая команда скорости.
