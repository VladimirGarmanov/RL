# Архитектура: от падений к плавной ходьбе

## Проблема: робот всегда падал

**Старые логи (models/skills/*):**
```
ep_len_mean = 40–80 шагов из 1000 (< 1 сек до падения)
approx_kl = 0.2–0.3 (политика разносилась)
std = 1.8–2.0 (слишком шумно)
```

**Причины:**

1. **Несовместимые модели** — среда менялась (добавился фильтр действий, изменилась награда), а старые модели обучались в другой версии. `reward_version` не совпадал.

2. **Агрессивные гиперпараметры** — `n_steps=1024`, `batch_size=512`, `clip_range=0.15` давали шумные обновления, алгоритм разносило.

3. **Неправильная архитектура** — обучали 4 специалиста отдельно, каждый в режиме "forward", игнорирующем команды. Все четыре сети выучили одно и то же.

4. **Просмотр в неправильной среде** — `play.py` создавал среду в режиме forward с буграми, но модель обучалась в режиме command на ровном полу. Среда и модель не совпадали.

## Решение

### 1. Переделали гиперпараметры на консервативные

```python
PPO_PARAMS = {
    "n_steps": 2048,          # было 1024
    "batch_size": 1024,       # было 512
    "clip_range": 0.15,       # оставили, но теперь на сильно обученных моделях
    ...
}

# При загрузке старой модели перекрываем её старые параметры
LOAD_OVERRIDES = {
    "n_steps": 2048, "batch_size": 1024, ...
}
```

→ Крупнее батчи = менее шумные градиенты = плавнее обучение.

### 2. Увеличили сеть и количество шагов

```python
# Было:
net_arch = [256, 256]
TOTAL_TIMESTEPS = 3_000_000
SKILL_TIMESTEPS = 2_000_000

# Стало:
net_arch = [512, 256]
TOTAL_TIMESTEPS = 3_000_000      # simple forward
COMMAND_TIMESTEPS = 12_000_000   # universal command (~18М с доучиванием)
SKILL_TIMESTEPS = 4_000_000      # specialist with warm start
```

→ Больше параметров = более гибкая политика. Больше шагов = лучше выучена.

### 3. Одна универсальная сеть вместо 4 специалистов

**Старая архитектура (падала):**
```
python train.py forward  ← видит только (v_x, 0), падает
python train.py left     ← видит только (0, w_z), падает
python train.py right    ← видит только (0, -w_z), падает
python train.py backward ← видит только (-v_x, 0), падает
```
Каждая обучалась отдельно, в режиме forward, игнорирующем команду. Результат — неправильное обучение + падения.

**Новая архитектура (работает):**
```
python train.py --reward-mode command
↓
Одна сеть видит полный вектор команды (v_x, w_z) на каждом шаге
Обучается отрабатывать ВСЕ команды: вперёд, назад, влево, вправо, стоп
Поэтому команда всегда совпадает с тем, что сеть видела во время обучения
```

→ Консистентность: одна среда, одна сеть, одна задача.

### 4. Версионирование среды

```python
# centipede_env.py
REWARD_VERSION = 3

# train.py сохраняет в meta.json при обучении
{"reward_version": 3, "reward_mode": "command", ...}

# play.py отказывается грузить несовместимые модели
if meta.get("reward_version") != REWARD_VERSION:
    print("WARNING: модель обучена старой версией среды")
```

→ Никогда больше не загрузим модель, обученную в другой физике.

### 5. Правильный режим в play.py

```python
# Было:
env = make_centipede_env(n_segments, render_mode="rgb_array")
# → reward_mode="forward" по умолчанию, с буграми

# Стало:
env = make_centipede_env(n_segments, reward_mode=meta.get("reward_mode"),
                         terrain_roughness=0.0)
# → режим совпадает с моделью, пол ровный
```

→ Среда просмотра совпадает с обучающей.

## Результаты

**До:** `ep_len_mean = 40–80` (падает на первой секунде)
**После:** `ep_len_mean > 800` (стоит, ходит плавно, не падает)

## Как запустить

```bash
# Одна команда, обучение на ночь
python train_overnight.py
# ↓
# 18 млн шагов, ~15–20 часов
# ↓
python play.py
# ↑ полный контроль стрелочками, плавная ходьба
```
